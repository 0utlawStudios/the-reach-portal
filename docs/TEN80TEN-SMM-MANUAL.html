01

System Overview

The Ten80Ten Content Engine is a full-stack content operations platform built to manage the complete lifecycle of social media content. It replaces fragmented toolchains (spreadsheets, shared folders, group chats, manual posting) with a single system that controls ideation, review, approval, scheduling, and publication across six major platforms.

The system operates on a five-stage Kanban pipeline. Content enters as raw ideas. It moves through structured review gates where approvers accept or reject work with written feedback. Approved content is scheduled with a target date and time. At the scheduled moment, the automation engine extracts the post data, pulls assets from Google Drive, and publishes to the designated platforms without human intervention.

Every action is logged in an append-only audit trail. Every file lives in a governed storage layer. Every user operates within a role-scoped permission boundary. The system is designed for creative teams that need both creative freedom and operational control.

Architecture

The Content Engine is a server-rendered React application deployed on edge infrastructure. The frontend handles all user interaction, real-time state synchronization, and optimistic UI updates. The backend manages authentication, file operations, notification dispatch, and external service orchestration through API routes that execute server-side.

Content Pipeline

The pipeline is the operational core of the system. Every piece of content passes through a deterministic sequence of stages. Stage transitions are permission-gated: only users with the appropriate role can move content forward or backward. Each transition generates an immutable audit record.

Stage 4
Approved / Scheduled

Stage 5
Posted
↺ Revision Needed and Awaiting Approval loop until the approver is satisfied

Stage Definitions

StageSort OrderWho Moves ForwardDescription
Ideas Newest first Any creator Raw concepts. Incomplete checklist items, no schedule required. Brainstorming zone with no approval pressure.
Awaiting Approval Earliest deadline first Creator submits Content is review-ready. Checklist complete, assets attached, caption finalized. Waiting for an approver to accept or reject.
Revision Needed Closest deadline first Approver kicks back Approver rejected the content with written feedback. Creator must address the notes, attach updated assets if needed, and resubmit. A mandatory "Fix submitted" note is logged on resubmission.
Approved / Scheduled Earliest schedule first Approver only Content passed review. Scheduled date and time are locked. The automation engine picks up this post and handles publication.
Posted Most recent first Automation engine Published content. Archived in reverse chronological order. Any posted card can be cloned back to Ideas for repurposing.

Post Card Data

Each card in the pipeline carries a structured payload:

  • TitleRequired. Short descriptor for internal tracking.
  • CaptionFull post text with hashtags. Platform-validated before approval.
  • HookOpening line or attention-grabbing phrase for the post.
  • PlatformsArray of target platforms (Instagram, Facebook, LinkedIn, X, TikTok, YouTube).
  • Content TypeVideo, Image, Carousel, Reel, or Story.
  • Scheduled Date/TimePublication target. Timezone-aware per workspace setting.
  • Thumbnail URLCover image reference for the card preview.
  • ChecklistJSONB array of pre-flight items (thumbnail ready, caption done, CTA added, etc.).
  • NotesRevision history. Append-only log of feedback and fix submissions.
  • Source VaultJSONB object linking master assets and publishable files.
  • Media IDsArray of Google Drive file references attached to this post.

Content Calendar

The calendar view renders all scheduled posts on a date-axis timeline. Posts appear on their scheduled date with platform badges and content type indicators. Clicking any calendar entry opens the full post detail. The calendar respects the workspace timezone setting (default: Asia/Dubai, configurable per workspace).

Post Repurposing

Any card in any stage can be cloned back to the Ideas column through the repurpose action. The system creates a new card with a fresh ID and reset checklist, preserving the caption, hook, platforms, and media references from the original. This supports content recycling workflows where a single concept gets adapted across formats or platforms.

05

Workspace Model

The Content Engine operates on a multi-tenant architecture. Each workspace is a fully isolated environment with its own posts, team members, media assets, brand configuration, and audit trail. Data never leaks between workspaces. Database queries are scoped through the is_active_workspace_member() RLS helper function that validates the requesting user's membership and role before returning any rows.

  • IsolationRow-Level Security on every table. Workspace ID foreign key required.
  • MembershipMany-to-many via workspace_members table (user can belong to multiple workspaces).
  • TimezonePer-workspace timezone setting. Affects calendar rendering and scheduled post display.
  • SlugURL-safe unique identifier for workspace routing.
  • StatusMembers carry active/pending/suspended status within each workspace independently.

Asset Management

All media files are stored in Google Drive through a service account integration. The system uses the Google Drive API v3 with resumable upload protocol (RFC 7578) for reliable large-file transfers. Files are organized into a governed folder taxonomy and tracked in the database for usage auditing.

Folder Taxonomy

FolderContentsAccess
/thumbnailsPost cover images, preview graphicsAll team members
/raw-filesSource materials, PSDs, project filesCreators, admins
/media-libraryFinalized, approved assets ready for publicationAll team members

Upload Protocol

File uploads follow a three-phase resumable protocol that handles network interruptions and large file sizes:

// Phase 1: Initiate
POST /api/drive/upload { fileName, mimeType, folder, cardId }
// Returns: { uploadUri, isImage, driveFileName }

// Phase 2: Stream
PUT {uploadUri} // Binary chunks sent directly to Google Drive

// Phase 3: Finalize
POST /api/drive/finalize { uploadUri, fileName, cardId }
// Returns: { driveFileId, url, name }
// Writes media_assets row, links to post via used_in[]

Two-Tier Asset Model

Each post card maintains two distinct asset categories through the Asset Review Drawer:

▤

Master Assets

Source files, design links, Google Drive folders, and raw materials. The editable originals that the creative team works from.

▣

Publishable Files

Final approved media ready for posting. These are the exact files the automation engine pulls when publishing to social platforms.

Brand Kit

Each workspace has a singleton Brand Playbook document stored as a JSONB column. This is the single source of truth for brand voice, hashtag strategy, content pillars, hooks, and calls-to-action. Admins and Creative Directors can edit the playbook in real-time. All team members reference it when creating content.

  • TaglineCompany positioning statement.
  • WebsitePrimary domain for link-in-bio and CTA references.
  • Service AreaGeographic or industry scope.
  • Hashtag SetsCore, seasonal, engagement, and commercial hashtag groups.
  • Content PillarsThree primary content themes with descriptions.
  • HooksSample opening phrases for posts and videos.
  • CTAsCall-to-action templates (e.g., "DM us", "Link in bio", "Book a call").
  • Posting ScheduleRecommended days and times per platform.
  • Brand VoiceTone, personality, and language guidelines for all content.
08

Collaboration Layer

Comments and Mentions

Every post card has a threaded comment section. Team members can tag others using @First Last syntax. The mention parser extracts names, resolves them against the team directory, and triggers an email notification to the mentioned user with a direct link back to the post.

Mention notifications are rate-limited to 10 per minute per IP address to prevent abuse. The email template includes the author name, post title, comment content, and a one-click link to the portal.

Revision Feedback Loop

When an approver kicks a post back to Revision Needed, the system captures their written feedback, attaches an optional reference URL, and sends a notification email to the content creator. The creator addresses the feedback, submits a "Fix submitted" note (required for resubmission), and the card moves back to Awaiting Approval. This loop continues until the approver is satisfied.

Real-Time Sync

The system uses Supabase Realtime channels to push database changes to all connected clients. When one user moves a card, edits a comment, or updates the brand kit, every other user sees the change within seconds. A deduplication layer (2-second TTL mutation set) prevents echo effects from the user's own actions.

Team Onboarding

The invite flow is a four-step process:

1

Admin Creates Invite

Admin enters the new member's email, name, and assigned role. The system creates an auth record and generates a secure invite link.

2

Email Delivered

A branded HTML email with the confirmation link is sent via SMTP. If email delivery fails, the admin receives a fallback URL to share manually.

3

User Completes Setup

The new member sets their password, uploads a profile photo (with circular crop), and enters their WhatsApp number.

4

Auto-Activation

On first login, the system transitions the member's status from pending to active and populates their profile from the team directory.

09

Automation Engine

The Content Engine runs on n8n, a self-hosted workflow automation platform. The n8n instance continuously monitors the pipeline and executes actions based on post state and schedule data. This removes manual posting from the workflow entirely.

Auto-Publish Workflow

When a post card reaches the Approved / Scheduled stage, the automation engine takes over. No human action is required beyond the approval itself.

1

Schedule Detection

n8n polls the posts table for cards in the approved_scheduled stage where scheduled_date and scheduled_time match the current window.

2

Payload Extraction

The automation engine reads the post's full payload: caption, hashtags, hook, target platforms, content type, and the source_vault JSONB object that references publishable files.

3

Asset Retrieval

The engine pulls the publishable media files from Google Drive using the stored file IDs. Assets are downloaded and prepared in the format required by each target platform.

4

Multi-Platform Publication

The post is published to each selected platform (Instagram, Facebook, LinkedIn, X, TikTok, YouTube) using their respective APIs. The caption, media, and metadata are formatted per-platform to respect character limits, aspect ratios, and file size constraints.

5

Stage Transition

On successful publication, the post card moves to the Posted stage. An audit log entry records the exact publication timestamp, platforms published, and automation execution ID. Publishable files are marked with an "Auto-Posted" badge in the Asset Review Drawer.

System Health Monitoring

n8n runs a daily health check at 7:00 AM (Asia/Manila) that probes all critical integrations. The check hits the /api/health/deep-check endpoint, which tests database connectivity, Google Drive API access, SMTP delivery, and n8n API availability. Results are compiled into an HTML email report sent to the operations team with pass/warn/fail status for each subsystem.

AI Content System

Autonomous Content Generation

The AI Content System transforms the pipeline from a human-operated workflow into a supervised autonomous loop. The AI generates complete, publish-ready content (graphics, video edits, captions, and hashtags) and places it directly into the Awaiting Approval stage. The only required human action is a binary decision: approve or send to revision.

How It Works

1

Content Generation

The AI engine reads the workspace's Brand Playbook (content pillars, voice, hashtag strategy, posting schedule) and generates complete post packages. Each package includes a platform-optimized graphic or video, a caption with hashtags, a hook line, and a CTA pulled from the brand kit. Generated assets are uploaded to Google Drive and linked to a new post card.

2

Pipeline Injection

The AI places each generated card directly into Awaiting Approval with a pre-filled checklist, scheduled date/time based on the brand's posting calendar, and all publishable files attached. The card is indistinguishable from human-created content in the pipeline.

3

Human Gate

An approver reviews the AI-generated content. Two outcomes:

Approve The card moves to Approved/Scheduled. The n8n engine handles publication at the scheduled time.

Revision Needed The approver writes specific feedback (e.g., "change the CTA", "wrong color palette", "shorten caption"). The card moves to Revision Needed with the feedback attached.

4

Autonomous Revision

The AI reads the revision feedback from the card's notes field, interprets the requested changes, regenerates or modifies the affected assets, updates the caption or creative as needed, logs a "Fix submitted" note, and moves the card back to Awaiting Approval. The revision loop continues until the human approver is satisfied.

5

Publication

Once approved, the n8n automation engine takes over. It extracts the post data, pulls the finalized assets from Google Drive, and publishes to the scheduled platforms at the exact scheduled time. The card moves to Posted. The full loop, from AI generation to live publication, requires only a single human approval click.

The AI-to-publish pipeline reduces the human role to quality control. Content ideation, creation, revision, and publication are handled by the system. The approver's job is to confirm that the output meets brand standards. One click to approve. One click to request changes. Everything else is automated.

Security and Audit

Data Access Control

Every database table is protected by PostgreSQL Row-Level Security policies. The is_active_workspace_member() function validates that the authenticated user holds an active membership in the target workspace with the required role before any read or write operation proceeds. Queries that fail this check return zero rows. There is no application-level bypass.

Audit Trail

The post_audit_logs table is append-only. No authenticated user can INSERT, UPDATE, or DELETE rows. Only the service role (server-side API routes) can write to this table. Every stage transition, mention notification, revision submission, and team invite generates an audit entry with a timestamp, acting user, action type, and detail payload.

Email Security

  • XSS PreventionAll user-generated content is HTML-escaped before email template injection.
  • Header InjectionSubject lines are sanitized to strip newlines and control characters.
  • AuthenticationSMTP uses app-specific passwords, not account credentials.

Rate Limiting

The rate_limit_buckets table implements token-bucket rate limiting. Mention notifications are capped at 10 per minute per IP address. The bucket refills on a rolling window with configurable token count and TTL.

Infrastructure

  • HostingVercel edge network with automatic TLS. No self-managed certificates.
  • DatabaseSupabase managed PostgreSQL with automated backups and point-in-time recovery.
  • SecretsAll API keys, service account credentials, and SMTP passwords stored in server-side environment variables. Never exposed to the client bundle.
  • Auth TokensJWT-based sessions with Supabase Auth. Tokens expire and refresh automatically.
